# HCC 记忆系统：设计哲学与架构思想

> **作者**: Wei
> **日期**: 2026-03-20
> **定位**: 本文档阐述 HCC 记忆系统的设计初心、核心理念和长期愿景。
> 它不是实现规格（那是 Plugin Spec 的职责），而是回答"为什么这样设计"。

---

## 1  问题的起源

编程智能体（AI Agent）在辅助仿真工作时，面临一个根本性的缺陷：**没有持续的记忆**。
每次会话结束，Agent 积累的所有上下文、犯过的错误、找到的解决方案、对物理现象的理解——
全部消失。下次开始时，它又是一张白纸。

这和人类工程师的工作方式形成了鲜明对比。一个有经验的 CFD 工程师之所以高效，
不是因为他比新手更聪明，而是因为他有**记忆**——他记得哪些配置容易出问题，
他记得某类流动的网格要怎么处理，他记得上次类似的工况用了什么方案。
这些记忆让他不需要每次都从头摸索。

HCC（Hierarchical Cognitive Caching）记忆系统的设计目标就是：
**给 Agent 补上这个缺失的能力——让它能像有经验的工程师一样，
从工作中积累、提炼、复用知识。**

---

## 2  核心理念：知识与记忆是两种不同的东西

在设计过程中，我们发现一个关键的概念区分：

### 知识（Knowledge）—— 抽象的、可共享的、相对稳定的

知识是已经被验证、被整理、被抽象过的经验总结。
它脱离了具体的人、具体的项目、具体的时间点。

举例：
- "OpenFOAM 中 simpleFoam 的初场不能全零，否则会导致浮点异常"
- "对于跨声速流动，对流项建议使用 linearUpwindV 格式"
- "k-omega SST 模型在强逆压梯度下比 k-epsilon 更准确"

这些是**可以写进教材的东西**。任何人读了都能理解，不需要知道是谁、在什么项目中总结出来的。

### 记忆（Memory）—— 具体的、个人的、动态的

记忆是在具体工作过程中产生的、带有强烈个人和项目特征的经验。

举例：
- "今天我在跑 LS89 算例时，初场全零导致了 FPE，通过 potentialFoam 解决了"
- "这个算例我把 relaxationFactors 从 0.7 降到 0.3 才收敛，因为入口湍流度设太高了"
- "我发现 blockMesh 的 simpleGrading 在前缘附近不够用，需要换成 edgeGrading"

这些信息**包含了"我"的痕迹**——我的算例、我的选择、我当时遇到的具体状况。
对我自己的后续工作非常有价值，但对别人来说，大部分价值不大。

### 两者的关系：从下往上的提炼

记忆和知识不是割裂的，它们之间有一条自然的**提炼路径**：

```
具体工作经验（大量、个人化、杂乱）
    ↓  提炼、去掉个人痕迹、验证可复现性
本地项目经验（较少、结构化、经过初步验证）
    ↓  跨项目验证、与权威参考交叉确认
可共享的知识（少量、高质量、通用性强）
    ↓  社区验证、多人确认
知识库中的条目（极少量、高度可靠、长期有效）
```

这个路径的关键特征是：**每一层都比上一层少，但质量更高。**
大量的日常记忆中，只有少数值得提炼为项目经验；
多个项目的经验中，只有少数值得共享到社区；
社区提交的知识中，只有少数能通过严格验证进入知识库。

**这就是 HCC 的"层级认知缓存"名称的由来**——
像 CPU 缓存一样，越靠近计算核心（Agent 的当前工作）的信息越多但越易变，
越远离核心的信息越少但越稳定可靠。

## 3  验证深度：不是所有"验证过的"记忆都同样可信

上一节描述了记忆从 L1 到 L3 的提炼路径——按"验证程度"分层。
但还有另一个同样重要的维度：**验证的深度**。

对于 CFD 仿真来说，"这条经验被验证了"这句话可以有四种完全不同的含义。
这四种含义之间存在一条递进的验证链，每一层都依赖于前一层，
但每一层能提供的信心也递增。

### 第一层：语法验证——"程序不报错了"

这是最基本的验证：修改了某个配置后，OpenFOAM 不再报 FOAM FATAL ERROR，
程序能够正常启动并运行。反馈信号是二值的（崩溃或不崩溃），
修复路径通常是确定的（错误信息直接指向了问题所在）。

这一层的记忆置信度天然较高，因为"不报错"是一个无歧义的标准。
迁移性也好——同样的语法错误在不同项目中的修复方法几乎总是一样的。

### 第二层：数值验证——"解收敛了"

程序不报错之后，下一个问题是：解收敛了吗？
残差是否在下降？质量/动量/能量守恒是否满足？
CFL 条件是否稳定？

这一层的反馈信号比语法层丰富得多（残差曲线、守恒误差、CFL 分布），
但也更加模糊——残差下降到 1e-4 算收敛吗？还是要到 1e-6？
不同的求解器、不同的物理问题，标准可能完全不同。

**这一层包含一个关键的认知陷阱：收敛不等于正确。**

一个只积累了大量数值记忆但缺少物理记忆的 Agent，
会变成一个**"擅长让错误的解收敛"的系统**。
它可能学会了一套让残差漂亮地下降到机器精度的松弛因子组合，
但得到的流场可能是非物理的——在应该有分离的地方没有分离，
激波位置偏了 20%，壁面换热系数差了一个量级。

更危险的是，这种 Agent 看起来很"有经验"（它的配置总能让计算收敛），
会给用户虚假的信心。**所以数值层的记忆必须附带一个警告：
"收敛不代表正确，仍需物理验证。"**

### 第三层：物理验证——"解符合物理"

这是记忆系统中**价值最高但积累最难**的层级。

要判断仿真结果是否"符合物理"，需要一个外部基准来闭合反馈回路——
实验数据、解析解、或高保真计算结果（如 DNS/LES 对 RANS 的校验）。
没有这个外部基准，"物理正确"就只是一个主观判断。

而且，即使有了外部基准，判断"符合物理"也不是一个简单的二值问题。
壁面压力分布可能在大部分区域吻合良好但在分离区偏差 20%——
这算"正确"还是"不正确"？答案取决于你关心的物理现象是什么。

因此，物理层的记忆提炼需要一个**结构化的验证-诊断-归因流程**：

1. **验证**：与基准数据对比哪些物理量？误差范围多大？
2. **诊断**：偏差出现在哪些区域？是系统性的还是局部的？
3. **归因**：偏差的原因是网格不够密、湍流模型不适合、
   边界条件设置不当、还是物理建模本身的局限？

只有经过这个流程提炼出的记忆，才真正值得进入 L3（wisdom）。
因为它不仅记录了"结果对不对"，还记录了"为什么对/不对"——
这个"为什么"才是可迁移的物理洞察。

### 第四层：方法论验证——"什么时候用什么方法"

这是最高层的抽象。它不是对某一次仿真结果的验证，
而是对一种工作方法在多个不同案例中的有效性验证。

典型的方法论记忆：
- "旋转机械先用 MRF 做稳态初场，再切换滑移网格做非定常"
- "对于叶栅流动，先在粗网格上跑出基本流场，再插值到细网格上继续"
- "含共轭传热的问题，先单独跑流体域到收敛，再耦合固体域"

这类记忆的特点是：它只能从多个成功项目的**共性模式**中涌现——
你不可能从一次仿真中总结出"旋转机械应该这样做"，
你需要在多个不同的旋转机械案例中都验证这个方法有效。

方法论记忆的迁移价值最高（一条好的方法论可以节省新手几周的摸索），
但验证也最难（需要在足够多的、足够多样的案例中确认普适性）。

### 四层之间的关系

这四层不是平行的分类，而是一条**递进的验证链**：

```
语法 → 数值 → 物理 → 方法论
(不报错)  (收敛了)  (结果对了)  (方法通用)
```

每一层都建立在前一层的基础之上——你不可能在程序还报错的时候讨论收敛性，
也不可能在解还没收敛的时候讨论物理正确性。

更重要的是，**每一层的提升都需要额外的信息来源**。
语法验证只需要程序本身的输出。
数值验证需要残差和守恒量的监控。
物理验证需要外部基准数据。
方法论验证需要跨案例的经验积累。

这意味着记忆的验证深度不可能被"自动"提升——
Agent 不可能仅通过自己的工作就完成所有四层验证。
它需要用户提供实验数据（物理层），需要在多个项目中被使用（方法论层）。
**这种"不完全自动"恰恰是设计上有意为之的**——
因为错误的高层验证比没有验证更危险。

## 4  通用知识与专项知识

除了验证深度之外，记忆还有另一个横向的维度：**通用性**。

### 通用知识（Foundation Knowledge）

与具体应用场景无关的底层知识。
比如 OpenFOAM 的常见报错和修复、网格质量指标的含义和阈值、
湍流模型选择的一般原则、fvSchemes 中各离散格式的适用场景。

这一层的特点是：条目数量相对少（可能 100-300 条就能覆盖很好），
但每条都需要非常高的质量和验证标准。
错误的通用知识造成的危害远大于错误的专项知识。

### 专项知识（Specialist Knowledge）

绑定在特定应用领域上的知识——叶轮机械、芯片热管理、汽车外气动等。
每个专项包含该领域特有的网格策略、边界条件选择、
物理现象理解、典型工况的配置模板、实验对比的经验。

**这个分法模拟了人类工程师的知识结构**。
一个做了十年叶轮机械仿真的工程师，他的大脑里有两层知识：
底层是所有 CFD 工程师都有的通用功底（求解器操作、基本物理、网格原则），
上层是叶轮机械的专业知识（叶栅流动、激波干涉、二次流、密封间隙等）。
当他转去做芯片冷却时，底层知识不需要重新学，只需要补充上层的专项知识。

### 判断一条经验属于哪一层

一个简单的判断方法：**如果把这条经验告诉一个做完全不同领域仿真的人，
他能用上吗？** 如果能，就是通用知识。如果不能，就是专项知识。

## 5  知识库与记忆系统的关系

基于上述分析，整个系统拆分为两个协作的子系统：

### 知识库（Knowledge Base）—— 只读的参考资料

知识库是**预先整理好的、可下载的、版本化的知识集合**。
它按 foundation / specialist 组织，
每个层的维护者和消费者都有明确的分工。

### 记忆系统（Memory System）—— 读写的工作伴侣

记忆系统是 **Agent 在工作过程中自动积累的个人经验**。
它是 HCC 三层架构（L1/L2/L3）的主体。

两者的交互：

**向下查询**（从知识库中获取参考）：
当 Agent 遇到问题时，先搜索本地记忆（L2/L3），
然后搜索已安装的知识库（通用 → 当前专项 → 其他专项）。

**向上贡献**（从记忆中产出新知识）：
当本地 L3 中的某条经验被认为具有通用价值时，
用户可以将其导出并提交到知识库。

```
Cloud Knowledge Base
  ├── Foundation (通用知识)
  └── Specialist (专项知识)
        │
        │  hcc pull (下载安装)
        │  hcc push (贡献上传)
        ▼
Local Project
  ├── knowledge/          ← 安装的知识库（只读参考）
  │   ├── foundation/
  │   └── specialist/
  │
  └── memory/             ← 本地记忆（读写，Agent 工作产物）
      ├── trace.md        (L1: 工作记忆)
      ├── findings/       (L2: 提炼经验)
      └── wisdom/         (L3: 验证知识 → 可能贡献到知识库)
```

## 6  记忆的生命周期

一条信息从产生到（可能）成为社区知识的完整旅程：

### 阶段一：产生（L1）

Agent 在工作过程中遇到了一个问题，经过调试解决了。
这个过程被 5-Action Rule 自动记录到 trace.md 中。

此时它只是一条流水账——"10:30 运行报错，10:45 检查初场，11:00 用 potentialFoam 解决"。

### 阶段二：提炼（L1 → L2）

用户或 Agent 执行 promote 操作，从流水账中提炼出结构化的经验。
提炼时，Agent 需要判断三个维度：

**Scope**：这条经验是项目特有的（@project）还是具有领域通用性的（@domain）？
**Layer**：这是通用知识（foundation）还是专项知识（specialist）？
**Validation Level**：这条经验被验证到了什么深度——
只是不报错了（syntax）？还是收敛了（numerical）？
还是跟实验数据对比过了（physical）？

**大部分记忆的 validation_level 在首次 promote 时停在 syntax 或 numerical。
这完全正常。** 它们作为本地经验在当前项目中发挥作用，
下次遇到类似问题时 Agent 可以直接复用。
但 Agent 和用户都应该清楚地知道：一条只通过了数值验证的配置经验，
**不等于**一条经过物理验证的可靠知识。

### 阶段三：深化验证（L2 内部升级）

随着项目推进，一条 finding 的验证深度可以逐步提升：
计算完成后做了后处理 → validation_level 从 syntax 升为 numerical。
和实验数据做了对比 → 从 numerical 升为 physical。
在另一个不同类型的案例中也验证有效 → 可能升为 methodology。

每次验证深度提升时，Evidence 和 Validation 区块都应该被更新，
记录新的验证证据。**验证深度的提升不是自动的——它需要用户提供外部信息
（实验数据、文献参考）或跨项目的工作经验。**

### 阶段四：升级（L2 → L3）

一条 finding 升级到 wisdom 的门槛取决于其类型和验证深度：

错误修复（EF）升级到 wisdom 最低要求 syntax 级别验证——
错误消失了就足以证明修复是有效的。

配置模式（CP）升级到 wisdom 最低要求 numerical 级别验证，
**强烈建议有 physical 级别**——一个"只让解收敛但不保证正确"的配置
进入 wisdom 是危险的，因为它会被后续使用者当作可信的参考。

物理洞察（PI）升级到 wisdom **必须**有 physical 级别验证——
物理洞察如果没有物理验证，它就只是一个猜测，不应该被当作可靠知识。

工作流（WF）升级到 wisdom 最低要求 numerical 级别验证。

### 阶段五：评估共享价值

用户审视 L3 中的条目，判断它是否值得共享到知识库。
判断标准：scope 是否为 @domain？去掉项目特定信息后是否仍有价值？
验证深度是否足够高？

### 阶段六：脱敏和导出

执行 export，系统自动去除个人信息，检查必填字段完整性，
输出符合知识库格式的标准化文件。

### 阶段七：社区审核和入库

提交到知识库，核心维护者审核格式、验证证据、检查重复。
通过审核后正式成为知识库的一部分。

**能走完这个完整旅程的信息是极少数。这是对的。**

## 7  为什么这个设计是 Agent 无关的

HCC 的核心机制——三层记忆、5-Action Rule、promote/complete 流程、
findings 条目格式——这些都不依赖任何特定的 Agent 实现。
它们是文件协议（Markdown）+ 操作协议（Bash 脚本）的组合。

当前的 MVP 选择以 Claude Code Plugin 的形式分发，
是因为 Claude Code 的 plugin 系统提供了最好的开箱即用体验
（斜杠命令、hooks、skills 自动发现）。

但核心的 scripts/ 目录中的 Bash 脚本可以被任何 Agent 调用。
未来如果需要适配 Cursor、Codex、Gemini CLI，
只需要为它们编写各自的"命令层"（.cursorrules、AGENTS.md 等），
底层的记忆管理逻辑完全复用。

**记忆数据本身（memory/ 目录中的 Markdown 文件）更是完全 Agent 无关的。**
不管是哪个 Agent 创建的 finding 条目，另一个 Agent 都能读取和使用。

## 8  长期愿景

### 阶段一：单人单项目（当前 MVP）

一个工程师在一个项目中使用 HCC Memory Plugin。
Agent 自动记录工作、提炼经验、积累验证知识。
跨会话的记忆恢复让每次工作都站在上次的肩膀上。

### 阶段二：单人多项目

同一个工程师在不同项目中使用 HCC。
每个项目有独立的 memory/，但可以通过 export/import
在项目之间迁移通用性强的经验。

### 阶段三：社区知识协作

多个工程师各自在自己的项目中积累记忆，
将最有价值的部分贡献到共享知识库。
知识库按 foundation/specialist 组织，
由各领域专家维护。

### 阶段四：知识的新陈代谢

随着求解器更新、最佳实践演化，
旧知识被验证过期、新知识被贡献入库。

### 阶段五：跨领域扩展

HCC Memory 作为通用的记忆基础设施，
不仅服务于 CFD 仿真，也可以服务于其他需要长期经验积累的领域。

## 9  设计原则总结

**记忆优先于知识。** 系统的首要价值是帮助单个 Agent 在单个项目中更好地工作。
社区知识库是加速器，不是地基。

**验证深度必须透明。** 每条记忆都必须标注它被验证到了什么程度——
语法、数值、物理、方法论。使用者必须能一眼看到这条记忆的可信度边界。
**一个知道自己哪些知识确定、哪些只是"看起来可以"的 Agent，
比一个盲目自信的 Agent 安全得多。**

**警惕"收敛≠正确"的陷阱。** 系统在设计上必须防止 Agent 变成一个
"擅长让错误的解收敛"的系统。数值级别的记忆必须附带警告，
物理级别的验证必须有外部基准支持。

**少而精优于多而杂。** 每一层的筛选都应该是严格的。
宁可漏掉一些有价值的记忆，也不要让低质量的信息污染知识库。

**从下往上而非从上往下。** 知识不是由某个权威"灌注"给用户的，
而是由社区中每个人的实际工作经验自然产出的。

**轻量可维护。** 纯 Markdown + Bash，无数据库依赖。

**开放可演进。** schema_version 字段确保未来可以升级格式，
用户已有的记忆不应该因为升级而丢失。
